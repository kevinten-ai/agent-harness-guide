# 安全与护栏

## 为什么安全如此关键

Agent 拥有真实的工具——它可以读写文件、执行代码、访问网络、操作 Git。一个出错的 Agent 可能：
- 删除重要文件
- 推送错误代码到生产环境
- 泄露敏感信息
- 无限循环消耗资源
- 执行非预期的破坏性操作

**安全不是可选项，它是 Harness 的基础设施。**

## 纵深防御: 五层独立安全层

核心思想：**任何单独一层都不被完全依赖**。每层都可能被绕过，但五层叠加的被绕过概率极低。

```
┌──────────────────────────────────────────────────┐
│ Layer 1: Prompt Guardrails (提示词护栏)           │
│ ─────────────────────────────────────────         │
│ 位置: System Prompt 中                            │
│ 方式: 用自然语言定义规则                           │
│                                                  │
│ 示例:                                            │
│ • "NEVER run rm -rf or git push --force"         │
│ • "Always read a file before editing it"         │
│ • "Never commit .env or credential files"        │
│                                                  │
│ 优势: 灵活、易理解                                │
│ 局限: 模型可能在长对话中"忘记"                     │
│ 加固: System Reminders 定期重注入                 │
└──────────────────────────────────────────────────┘
                      │
                      ▼
┌──────────────────────────────────────────────────┐
│ Layer 2: Schema Restrictions (Schema 级限制)      │
│ ─────────────────────────────────────────         │
│ 位置: 工具注册层                                  │
│ 方式: 不在 Schema 中的工具 = 不可调用              │
│                                                  │
│ 应用:                                            │
│ • Plan Mode: 只注入只读工具 Schema                │
│ • SubAgent: 按角色限制可用工具                     │
│ • 权限级别: 不同信任级别暴露不同工具               │
│                                                  │
│ 优势: 绝对可靠 (没有 Schema → 模型无法生成调用)   │
│ 局限: 粒度较粗 (整个工具级别)                     │
└──────────────────────────────────────────────────┘
                      │
                      ▼
┌──────────────────────────────────────────────────┐
│ Layer 3: Runtime Approval (运行时审批)            │
│ ─────────────────────────────────────────         │
│ 位置: 工具调度前                                  │
│ 方式: 模式匹配 + 人工确认                         │
│                                                  │
│ 三个级别:                                        │
│ ┌──────────────────────────────────────┐         │
│ │ Manual    (手动): 每次工具调用都需确认 │         │
│ │ Semi-Auto (半自动): 模式匹配自动放行,  │         │
│ │           未匹配的需确认              │         │
│ │ Auto      (全自动): 全部自动放行      │         │
│ └──────────────────────────────────────┘         │
│                                                  │
│ 模式匹配示例:                                     │
│ • Allow: Read(*), Glob(*), Grep(*)               │
│ • Allow: Bash("npm test"), Bash("npm run *")     │
│ • Deny:  Bash("rm *"), Bash("*--force*")         │
│ • Ask:   其他所有                                 │
└──────────────────────────────────────────────────┘
                      │
                      ▼
┌──────────────────────────────────────────────────┐
│ Layer 4: Tool Validation (工具级校验)             │
│ ─────────────────────────────────────────         │
│ 位置: 工具执行内部                                │
│ 方式: 参数校验 + 危险模式检测                      │
│                                                  │
│ 检查项:                                          │
│ • 危险命令黑名单 (rm -rf /, dd if=, ...)         │
│ • 过期读取检测 (文件已被修改但读取的是旧版)        │
│ • 执行超时 (默认 2min, 最大 10min)               │
│ • 输出大小限制                                    │
│ • 命令注入防御 (转义特殊字符)                      │
│ • Edit 前必须 Read (防止盲改)                     │
│                                                  │
│ 优势: 即使前三层都被绕过,这层仍然生效             │
└──────────────────────────────────────────────────┘
                      │
                      ▼
┌──────────────────────────────────────────────────┐
│ Layer 5: Lifecycle Hooks (生命周期钩子)           │
│ ─────────────────────────────────────────         │
│ 位置: Agent 执行的关键节点                        │
│ 方式: 用户自定义的拦截/验证逻辑                    │
│                                                  │
│ Hook 事件:                                       │
│ ┌──────────────────────────────────────┐         │
│ │ PreToolUse   → 工具调用前拦截/修改    │         │
│ │ PostToolUse  → 工具调用后验证         │         │
│ │ Stop         → 任务结束时审计         │         │
│ │ SessionStart → 会话开始时初始化       │         │
│ │ SessionEnd   → 会话结束时清理         │         │
│ │ UserPromptSubmit → 用户输入前处理     │         │
│ │ Notification → 通知事件              │         │
│ └──────────────────────────────────────┘         │
│                                                  │
│ 能力:                                            │
│ • 阻止: 返回错误阻止工具执行                      │
│ • 修改: 修改工具参数                              │
│ • 审计: 记录所有操作                              │
│ • 通知: 发送告警                                  │
└──────────────────────────────────────────────────┘
```

## 操作风险分级

```
┌────────────────────────────────────────────────────────┐
│ Risk Level    Examples              Action              │
│ ─────────    ──────────            ──────               │
│ LOW          Read, Glob, Grep      Auto-approve         │
│              (只读、本地、可逆)                           │
│                                                        │
│ MEDIUM       Edit, Write, Bash     Confirm first time   │
│              (本地写入,可逆)        then auto-approve    │
│                                                        │
│ HIGH         git push, API calls   Always confirm       │
│              (影响外部,难以撤回)                          │
│                                                        │
│ CRITICAL     rm -rf, force push    Block + warn         │
│              (破坏性,不可逆)        require explicit ask │
└────────────────────────────────────────────────────────┘
```

## 敏感数据保护

```
Agent 不应该:
├── 提交 .env 文件
├── 打印 API keys/tokens
├── 将凭证写入代码
├── 将敏感数据发送到外部服务
└── 在日志中包含密码

防护手段:
├── Git: .gitignore + 提交前检查
├── Prompt: "Never commit files containing secrets"
├── Hook: PreToolUse 检测敏感文件
├── Tool: Write 工具检查内容模式
└── Review: 提交前自动扫描
```

## Doom Loop 防护

```
检测机制:
┌─────────────────────────────────────────┐
│ Counter: per_file_edit_count            │
│ Threshold: 5 edits to same file         │
│                                         │
│ Counter: repeated_tool_call_pattern     │
│ Threshold: 3 identical calls            │
│                                         │
│ Counter: total_iterations               │
│ Threshold: configurable max (e.g. 100)  │
│                                         │
│ Action on threshold breach:             │
│ ├── 注入 "reconsider" 提示              │
│ ├── 建议尝试不同方法                    │
│ ├── 如果持续 → 强制停止                 │
│ └── 报告当前状态给用户                  │
└─────────────────────────────────────────┘
```

## 可逆性思维

```
核心原则: 优先选择可逆操作

操作前思考:
├── 这个操作可以撤销吗？
├── 影响范围有多大？(本地 vs 远程)
├── 如果出错，恢复成本是多少？
└── 有没有更安全的替代方案？

示例:
├── git reset --hard → 先 git stash (可恢复)
├── 删除文件 → 先确认未提交变更已保存
├── Force push → 几乎永远不应该自动执行
├── 数据库操作 → 先备份
└── API 调用 → 检查幂等性
```
