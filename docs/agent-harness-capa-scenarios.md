# Agent Harness + Capa 产品应用场景深度分析

## 产品愿景

**AI Agent 作为企业中间件的"智能中枢"**

通过 Agent Harness 提供 Agent 运行时基础设施，结合 Capa 统一中间件标准 API，使 AI Agent 能够直接操作所有企业中间件（Kafka、Redis、MySQL、配置中心、消息队列等），实现自然语言驱动的中间件运维、管理和优化。

## 场景一：故障排查场景

### 场景描述

用户通过自然语言描述故障现象，Agent 自动进行诊断，涉及日志查询、指标分析、配置检查、链路追踪等多维度排查，最终定位根因并提供修复建议。

### 目标用户
- SRE（站点可靠性工程师）
- 运维工程师
- 开发人员（自助排查）

### 用户故事

```
作为 SRE 工程师
当收到告警"订单服务响应时间超过 5s"时
我希望通过自然语言询问 Agent 进行诊断
以便快速定位是网络问题、数据库慢查询还是缓存失效导致的
```

### 详细流程

```
用户输入: "订单服务响应慢，帮我诊断一下"
      │
      ▼
┌─────────────────────────────────────────────┐
│ Step 1: 信息收集                             │
│ • 查询 Prometheus 获取订单服务 P99 延迟趋势    │
│ • 查询最近 30 分钟错误率变化                   │
│ • 获取当前 QPS 和连接数                       │
└──────────────┬──────────────────────────────┘
               │
               ▼
┌─────────────────────────────────────────────┐
│ Step 2: 关联分析                             │
│ • 查询链路追踪(Jaeger/SkyWalking)            │
│ • 找出耗时最长的调用链                        │
│ • 识别热点方法/慢 SQL                         │
└──────────────┬──────────────────────────────┘
               │
               ▼
┌─────────────────────────────────────────────┐
│ Step 3: 深度诊断                             │
│ • 查询 MySQL 慢查询日志                      │
│ • 检查 Redis 缓存命中率                       │
│ • 查看 Kafka 消息堆积情况                     │
│ • 检查配置中心是否有变更                      │
└──────────────┬──────────────────────────────┘
               │
               ▼
┌─────────────────────────────────────────────┐
│ Step 4: 根因定位                             │
│ Agent: "发现 MySQL 慢查询增加，具体 SQL:      │
│         SELECT * FROM orders WHERE...        │
│         缺少索引，导致全表扫描"               │
└──────────────┬──────────────────────────────┘
               │
               ▼
┌─────────────────────────────────────────────┐
│ Step 5: 修复建议                             │
│ • 生成优化后的 SQL 和索引建议                 │
│ • 提供临时缓解方案（如限流配置）              │
│ • 创建修复工单                               │
└─────────────────────────────────────────────┘
```

### 涉及的技术组件

| 组件类型 | 具体技术 | Capa API |
|---------|---------|----------|
| 指标存储 | Prometheus, VictoriaMetrics | Telemetry API |
| 链路追踪 | Jaeger, SkyWalking, Zipkin | Telemetry API |
| 日志系统 | ELK, Loki, ClickHouse | Telemetry API |
| 配置中心 | Nacos, Apollo, Consul | Config API |
| 数据库 | MySQL, PostgreSQL | State API / RPC |
| 缓存 | Redis, Memcached | State API |
| 消息队列 | Kafka, RocketMQ | PubSub API |

### 业务价值

- **时间节省**: 故障排查时间从 30-60 分钟缩短到 5-10 分钟
- **知识沉淀**: 诊断过程自动记录，形成知识库
- **降低门槛**: 初级工程师也能进行复杂故障诊断
- **预防复发**: 识别根因后自动生成监控规则

### 实现复杂度评估

**中**

主要挑战：
- 多源数据关联分析的逻辑设计
- 故障模式库的构建
- 诊断结论的准确性验证

## 场景二：配置变更场景

### 场景描述

用户通过自然语言发起配置变更请求，Agent 理解意图后执行完整的变更流程：审批、灰度发布、验证、全量发布、监控回滚。

### 目标用户
- 运维工程师
- 配置管理员
- 开发团队负责人

### 用户故事

```
作为运维工程师
当需要调整服务限流阈值时
我希望用自然语言告诉 Agent"将订单服务限流从 1000 调整到 2000"
Agent 自动完成审批、灰度、验证、全量发布的完整流程
```

### 详细流程

```
用户输入: "将订单服务的限流阈值从 1000 调整到 2000 QPS"
      │
      ▼
┌─────────────────────────────────────────────┐
│ Step 1: 意图理解与参数提取                   │
│ • 识别目标服务: order-service               │
│ • 识别配置项: rate_limit_threshold          │
│ • 识别变更: 1000 → 2000                     │
│ • 查询当前配置确认                           │
└──────────────┬──────────────────────────────┘
               │
               ▼
┌─────────────────────────────────────────────┐
│ Step 2: 风险评估                             │
│ • 查询该配置的历史变更记录                    │
│ • 分析影响范围（哪些服务依赖此配置）          │
│ • 评估风险等级: 中风险                        │
│ • 生成变更计划                               │
└──────────────┬──────────────────────────────┘
               │
               ▼
┌─────────────────────────────────────────────┐
│ Step 3: 审批流程                             │
│ • 创建变更工单                               │
│ • 根据风险等级路由审批人                      │
│ • 等待审批完成（或根据策略自动审批）          │
└──────────────┬──────────────────────────────┘
               │
               ▼
┌─────────────────────────────────────────────┐
│ Step 4: 灰度发布                             │
│ • 选择灰度节点（如 10% 实例）                │
│ • 通过 Config API 推送配置                   │
│ • 监控灰度节点指标（错误率、延迟）            │
│ • 持续 5 分钟无异常则继续                     │
└──────────────┬──────────────────────────────┘
               │
               ▼
┌─────────────────────────────────────────────┐
│ Step 5: 全量发布                             │
│ • 逐步全量推送配置                           │
│ • 实时监控全局指标                            │
│ • 记录配置版本快照                            │
└──────────────┬──────────────────────────────┘
               │
               ▼
┌─────────────────────────────────────────────┐
│ Step 6: 验证与监控                           │
│ • 对比变更前后关键指标                        │
│ • 生成变更报告                               │
│ • 设置异常自动回滚（如错误率 > 1%）          │
└─────────────────────────────────────────────┘
```

### 涉及的技术组件

| 组件类型 | 具体技术 | Capa API |
|---------|---------|----------|
| 配置中心 | Nacos, Apollo, Consul, etcd | Config API |
| 审批系统 | 自研/OA 系统 | RPC API |
| 工单系统 | Jira, 禅道, 自研 | RPC API |
| 监控系统 | Prometheus + Alertmanager | Telemetry API |
| 服务注册 | Nacos, Consul, Eureka | RPC API |

### 安全机制

```
┌─────────────────────────────────────────────┐
│ 配置变更安全护栏                             │
├─────────────────────────────────────────────┤
│ 1. 变更预审                                  │
│    • 敏感配置项白名单                         │
│    • 数值范围校验（如限流不能 > 10000）       │
│    • 正则匹配（防止 SQL 注入类配置）          │
│                                             │
│ 2. 权限控制                                  │
│    • 基于 RBAC 的配置项权限                   │
│    • 服务归属验证                             │
│                                             │
│ 3. 审计追踪                                  │
│    • 全量变更日志                             │
│    • 变更前后值对比                           │
│    • 操作人、时间、原因记录                   │
│                                             │
│ 4. 自动回滚                                  │
│    • 异常指标自动检测                         │
│    • 一键回滚到上一版本                       │
│    • 配置版本管理（保留最近 30 个版本）       │
└─────────────────────────────────────────────┘
```

### 业务价值

- **降低风险**: 灰度发布 + 自动回滚，避免配置错误导致故障
- **提升效率**: 配置变更从 30 分钟缩短到 5 分钟
- **合规审计**: 完整的变更记录，满足审计要求
- **知识传承**: 配置变更历史可查询，便于问题追溯

### 实现复杂度评估

**高**

主要挑战：
- 审批流程与现有 OA 系统集成
- 灰度策略的灵活配置
- 自动回滚的触发条件设计
- 多配置中心的统一抽象

## 场景三：容量规划场景

### 场景描述

Agent 分析历史资源使用数据，预测未来容量需求，自动触发扩缩容操作，确保服务容量始终匹配业务需求。

### 目标用户
- SRE 团队
- 运维负责人
- 成本优化团队

### 用户故事

```
作为 SRE 负责人
我希望 Agent 能够自动分析历史数据并预测容量需求
在业务高峰期前自动扩容，在低谷期自动缩容
以便在保证服务质量的同时优化成本
```

### 详细流程

```
┌─────────────────────────────────────────────┐
│ Step 1: 数据收集                             │
│ • 查询 Prometheus 获取历史指标               │
│   - CPU 使用率（过去 90 天，小时粒度）        │
│   - 内存使用率                               │
│   - QPS/TPS 趋势                             │
│   - 连接数、线程池使用率                      │
│ • 获取业务事件数据（促销活动、节假日）        │
└──────────────┬──────────────────────────────┘
               │
               ▼
┌─────────────────────────────────────────────┐
│ Step 2: 趋势分析                             │
│ • 识别增长趋势（线性/指数/周期性）            │
│ • 检测异常峰值和季节性模式                    │
│ • 计算资源使用与业务指标的关联性              │
└──────────────┬──────────────────────────────┘
               │
               ▼
┌─────────────────────────────────────────────┐
│ Step 3: 容量预测                             │
│ • 基于时序模型预测未来 30 天需求              │
│ • 考虑已知业务事件（如双 11）                 │
│ • 计算置信区间和安全水位                      │
│ • 输出: "预计 7 天后 CPU 将达到 85%"         │
└──────────────┬──────────────────────────────┘
               │
               ▼
┌─────────────────────────────────────────────┐
│ Step 4: 扩容建议                             │
│ • 生成扩容方案（实例数、规格）                │
│ • 成本估算                                   │
│ • 风险评估                                   │
└──────────────┬──────────────────────────────┘
               │
               ▼
┌─────────────────────────────────────────────┐
│ Step 5: 自动/半自动执行                      │
│ 场景 A: 自动扩容（已配置策略）                │
│   • 非高峰时段自动扩容                        │
│   • 逐步扩容，验证每一步                      │
│                                             │
│ 场景 B: 人工确认（重大变更）                  │
│   • 生成详细的扩容计划书                      │
│   • 等待人工审批后执行                        │
└──────────────┬──────────────────────────────┘
               │
               ▼
┌─────────────────────────────────────────────┐
│ Step 6: 效果验证                             │
│ • 监控扩容后资源使用率                        │
│ • 对比预测值与实际值                          │
│ • 持续优化预测模型                            │
└─────────────────────────────────────────────┘
```

### 涉及的技术组件

| 组件类型 | 具体技术 | Capa API |
|---------|---------|----------|
| 时序数据库 | Prometheus, InfluxDB, TDengine | Telemetry API |
| 预测模型 | 内置统计模型 / 外部 ML 服务 | RPC API |
| 容器平台 | Kubernetes, Docker Swarm | RPC API |
| 云服务 | AWS Auto Scaling, 阿里云 ESS | RPC API |
| 成本分析 | 云厂商账单 API | RPC API |

### 业务价值

- **成本优化**: 避免过度配置，预计节省 20-30% 资源成本
- **质量保证**: 提前扩容避免容量不足导致的服务降级
- **自动化**: 减少人工监控和决策成本
- **可预测性**: 提前 7-30 天预知容量需求

### 实现复杂度评估

**高**

主要挑战：
- 预测模型的准确性调优
- 业务事件与资源需求的关联分析
- 多维度指标的综合决策
- 自动扩容的安全边界设计

## 场景四：数据迁移场景

### 场景描述

Agent 协助完成跨存储的数据迁移任务，包括迁移计划制定、一致性校验、进度监控、回滚预案等。

### 目标用户
- DBA
- 数据工程师
- 架构师

### 用户故事

```
作为 DBA
我需要将用户表从 MySQL 迁移到 TiDB
希望 Agent 能够制定详细的迁移计划
包括数据校验、增量同步、回滚方案
确保迁移过程零数据丢失
```

- **保证一致性**: 多层校验确保数据零丢失

### 实现复杂度评估

**高**

主要挑战：
- 异构数据库的 Schema 映射

- 增量同步的实时性保证

## 关键指标概览
| 组件 | 健康度 | 关键指标 |
|------|--------|----------|
| MySQL-主库 | ✅ 正常 | CPU 45%, 连接数 120/500 |
| MySQL-从库 | ⚠️ 警告 | 主从延迟 2.3s (阈值 1s) |

| Kafka-订单 | ✅ 正常 | 消息堆积 0, 消费正常 |

- **根因**: 会话过期时间配置过长（30天）

3. Nacos 配置监听数过多，建议清理无效监听

### 代码生成示例

```java
// Agent 生成的 Redis 缓存接入代码示例

// 1. 添加依赖 (pom.xml)
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-data-redis</artifactId>
</dependency>

// 2. 配置文件 (application.yml)
spring:
  redis:
    host: ${REDIS_HOST:redis-cluster.internal}
    port: 6379
    password: ${REDIS_PASSWORD}
    lettuce:
      pool:
        max-active: 8
        max-idle: 8
        min-idle: 0

// 3. 缓存配置类
@Configuration
@EnableCaching
public class RedisConfig {

    @Bean
    public CacheManager cacheManager(RedisConnectionFactory factory) {
        return RedisCacheManager.builder(factory)
            .cacheDefaults(RedisCacheConfiguration.defaultCacheConfig()
                .entryTtl(Duration.ofMinutes(10))  // 默认过期时间
                .serializeKeysWith(RedisSerializationContext.SerializationPair
                    .fromSerializer(new StringRedisSerializer()))
                .serializeValuesWith(RedisSerializationContext.SerializationPair
                    .fromSerializer(new GenericJackson2JsonRedisSerializer())))
            .build();
    }
}

// 4. 使用示例
@Service
public class OrderService {

    @Cacheable(value = "order", key = "#orderId")
    public Order getOrder(String orderId) {
        // 缓存未命中时执行
        return orderRepository.findById(orderId);
    }

    @CacheEvict(value = "order", key = "#orderId")
    public void updateOrder(String orderId, Order order) {
        // 更新后清除缓存
        orderRepository.save(order);
    }
}
```
