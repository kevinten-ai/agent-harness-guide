# Phase 4: 生产实践

## 学习目标

将 Agent Harness 知识应用到实际生产场景，建立持续改进的实践。

## 主题 1: 评估与基准

### 建立评估体系

```
Agent 质量的衡量维度:

1. 效果指标
├── 任务完成率 (Task Completion Rate)
├── 首次正确率 (First-Time Correctness)
├── 回退率 (Rollback Rate)
└── 人工介入率 (Human Intervention Rate)

2. 效率指标
├── 平均迭代次数 / 任务
├── Token 使用量 / 成功任务
├── 平均任务耗时
├── Prompt Cache 命中率
└── API 调用次数 / 任务

3. 安全指标
├── 安全拦截次数
├── Doom Loop 触发率
├── 人工审批通过率
├── 敏感操作检出率
└── 误报率 (false positive)

4. 可靠性指标
├── Agent 崩溃率
├── 上下文溢出频率
├── 恢复成功率
└── 平均故障间隔时间 (MTBF)
```

### 回归测试

```
为 Harness 建立回归测试套件:

测试类型:
├── 工具测试: 每个工具的正确性
├── 安全测试: 纵深防御的每一层
├── 压缩测试: 上下文压缩后的质量
├── 恢复测试: 中断后的恢复能力
├── 边界测试: 极端输入的处理
└── 集成测试: 完整 Agent Loop 的端到端

自动化:
├── CI/CD 集成
├── 模型版本变更时自动运行
├── Harness 变更时自动运行
└── 定期基准测试
```

## 主题 2: 团队协作

### CLAUDE.md 最佳实践

```
CLAUDE.md 是团队与 Agent 的"合约":

推荐结构:
├── 项目概述 (Agent 需要知道的上下文)
├── 技术栈 (使用什么、不使用什么)
├── 代码规范 (命名、格式、结构)
├── 测试要求 (覆盖率、测试策略)
├── 部署流程 (CI/CD、环境)
├── 禁止事项 (绝对不能做的事)
└── 常见陷阱 (新人容易犯的错误)

维护原则:
├── 保持简洁 (Agent 也有上下文限制)
├── 定期更新 (与代码同步)
├── 只写"非显而易见"的规则
├── 不写能从代码推断的信息
└── 测试: 新 Agent 读 CLAUDE.md 后能否正确工作?
```

### Hook 和 Plugin 库

```
为团队建立共享的 Harness 组件:

安全 Hooks:
├── 敏感文件检测
├── 大文件写入告警
├── 生产环境操作拦截
└── API Key 泄露检测

质量 Hooks:
├── 提交前测试运行
├── 代码格式化检查
├── 类型检查执行
└── 依赖安全扫描

效率 Plugins:
├── 项目特定的 Skill
├── 团队工作流自动化
├── 常见任务模板
└── 代码生成模板
```

## 主题 3: 持续改进

### Trace-Based Improvement (基于追踪的改进)

```
循环: Observe → Analyze → Improve → Verify

1. 收集失败 Trace
├── Agent 失败的完整对话
├── Doom Loop 触发的上下文
├── 人工接管的原因
└── 超时或异常终止的场景

2. 分析模式
├── 什么类型的任务失败率最高?
├── 哪些工具调用最容易出错?
├── Agent 在哪些步骤卡住?
├── 人工介入通常是因为什么?
└── 有没有可预防的失败模式?

3. 产生改进
├── 更新 System Prompt (新规则)
├── 优化工具 Schema (更好的描述)
├── 添加新的安全检查
├── 改进上下文管理策略
└── 调整 Doom Loop 阈值

4. 验证效果
├── A/B 测试新旧 Harness
├── 运行回归测试套件
├── 监控关键指标变化
└── 确认改进不引入新问题

关键理念:
"每个 Agent 失败都应该产生一个永久的环境修复,
 而不仅仅是一次重试。"
```

### OpenAI 的 Entropy Management (熵管理)

```
概念: Agent 系统随时间积累"熵"(不一致性)

来源:
├── 代码风格漂移
├── 注释与代码不一致
├── 未使用的导入/变量
├── 不一致的命名
└── 过时的文档

解决: "垃圾回收" Agent

定期运行的 Agent:
├── 扫描整个代码库
├── 发现不一致和问题
├── 自动修复简单问题
├── 报告需要人工处理的问题
└── 更新文档和注释

效果: 保持代码库的整洁和一致
```
